# Hello, World!

이 챕터에서는 새 패키지를 만들고, 간단한 모듈을 작성하고, 컴파일하고, Move CLI로 테스트를 실행하는 방법을 배웁니다. [Sui를 설치](./../before-we-begin/install-sui)하고 [IDE 환경](./../before-we-begin/ide-support)을 설정했는지 확인하세요. 아래 명령어를 실행하여 Sui가 올바르게 설치되었는지 테스트하세요.

```bash
# 클라이언트 버전이 출력되어야 합니다. 예: sui-client 1.22.0-036299745.
sui client --version
```

> Move CLI는 Move 언어를 위한 명령줄 인터페이스입니다. Sui 바이너리에 내장되어 있으며 패키지 관리, 코드 컴파일 및 테스트를 위한 명령 세트를 제공합니다.

이 챕터의 구조는 다음과 같습니다:

- [새 패키지 만들기](#새-패키지-만들기)
- [디렉토리 구조](#디렉토리-구조)
- [패키지 컴파일하기](#패키지-컴파일하기)
- [테스트 실행하기](#테스트-실행하기)

## 새 패키지 만들기

새 프로그램을 만들려면 `sui move new` 명령어 뒤에 애플리케이션 이름을 입력합니다. 우리의 첫 번째 프로그램은 `hello_world`라고 하겠습니다.

> 참고: 이 챕터와 다른 챕터에서 `$`(달러 기호)로 시작하는 줄이 있는 코드 블록을 보면, 다음 명령어가 터미널에서 실행되어야 한다는 의미입니다. 기호는 포함하지 마세요. 이는 터미널 환경에서 명령어를 표시하는 일반적인 방법입니다.

```bash
$ sui move new hello_world
```

`sui move` 명령어는 Move CLI에 대한 액세스를 제공합니다 - 내장 컴파일러, 테스트 러너 및 Move와 관련된 모든 유틸리티입니다. `new` 명령어 뒤에 패키지 이름을 입력하면 새 폴더에 새 패키지가 생성됩니다. 우리의 경우 폴더 이름은 "hello_world"입니다.

폴더의 내용을 보면 패키지가 성공적으로 생성되었음을 확인할 수 있습니다.

```bash
$ ls -l hello_world
Move.toml
sources
tests
```

## 디렉토리 구조

Move CLI는 애플리케이션의 스캐폴드를 생성하고 디렉토리 구조와 필요한 모든 파일을 미리 생성합니다. 내부를 살펴보겠습니다.

```plaintext
hello_world
├── Move.toml
├── sources
│   └── hello_world.move
└── tests
    └── hello_world_tests.move
```

### 매니페스트

[패키지 매니페스트](./../concepts/manifest)로 알려진 `Move.toml` 파일은 패키지의 정의와 구성 설정을 포함합니다. Move 컴파일러가 패키지 메타데이터를 관리하고, 의존성을 가져오고, 명명된 주소를 등록하는 데 사용됩니다. [개념](./../concepts) 챕터에서 자세히 설명하겠습니다.

> 기본적으로 패키지에는 하나의 명명된 주소가 있습니다 - 패키지의 이름입니다.

```toml
[addresses]
hello_world = "0x0"
```

### 소스

`sources/` 디렉토리에는 소스 파일이 포함되어 있습니다. Move 소스 파일은 _.move_ 확장자를 가지며, 일반적으로 파일에 정의된 모듈의 이름을 따서 명명됩니다. 예를 들어, 우리의 경우 파일 이름은 _hello_world.move_이며 Move CLI는 이미 주석 처리된 코드를 내부에 배치했습니다:

```move
/*
/// Module: hello_world
module hello_world::hello_world;
*/
```

> `/*`와 `*/`는 Move의 주석 구분자입니다. 그 사이의 모든 것은 컴파일러에 의해 무시되며 문서나 메모에 사용할 수 있습니다. [기본 구문](./../move-basics/comments)에서 코드에 주석을 다는 모든 방법을 설명합니다.

주석 처리된 코드는 모듈 정의입니다. `module` 키워드로 시작하고 그 뒤에 명명된 주소(또는 주소 리터럴)와 모듈 이름이 옵니다. 모듈 이름은 모듈의 고유 식별자이며 패키지 내에서 고유해야 합니다. 모듈 이름은 다른 모듈이나 트랜잭션에서 모듈을 참조하는 데 사용됩니다.

{/* 그리고 모듈 이름은 유효한 Move 식별자여야 합니다: 단어를 구분하기 위해 밑줄을 사용하는 영숫자. 일반적인 규칙은 모듈(및 함수)을 snake_case로 호출하는 것입니다 - 모두 소문자에 밑줄 사용. 코딩 규칙은 코드의 가독성과 유지 관리성을 위해 중요하며, 코딩 규칙 섹션에서 요약합니다. */}

### 테스트

`tests/` 디렉토리에는 패키지 테스트가 포함되어 있습니다. 컴파일러는 일반 빌드 프로세스에서 이러한 파일을 제외하지만 _test_ 및 _dev_ 모드에서는 사용합니다. 테스트는 Move로 작성되며 `#[test]` 속성으로 표시됩니다. 테스트는 별도의 모듈로 그룹화할 수 있으며(이 경우 일반적으로 _module_name_tests.move_라고 함), 또는 테스트하는 모듈 내부에 있을 수 있습니다.

모듈, 임포트, 상수 및 함수는 `#[test_only]`로 주석 처리할 수 있습니다. 이 속성은 빌드 프로세스에서 모듈, 함수 또는 임포트를 제외하는 데 사용됩니다. 이는 체인에 게시될 코드에 포함시키지 않고 테스트를 위한 헬퍼를 추가하려는 경우에 유용합니다.

_hello_world_tests.move_ 파일에는 주석 처리된 테스트 모듈 템플릿이 포함되어 있습니다:

```move
/*
#[test_only]
module hello_world::hello_world_tests;
// 이 줄의 주석을 해제하여 모듈을 임포트하세요
// use hello_world::hello_world;

const ENotImplemented: u64 = 0;

#[test]
fun test_hello_world() {
    // 통과
}

#[test, expected_failure(abort_code = hello_world::hello_world_tests::ENotImplemented)]
fun test_hello_world_fail() {
    abort ENotImplemented
}
*/
```

### 기타 폴더

추가로 Move CLI는 `examples/` 폴더를 지원합니다. 여기에 있는 파일은 `tests/` 폴더에 배치된 파일과 유사하게 처리됩니다 - _test_ 및 _dev_ 모드에서만 빌드됩니다. 패키지 사용 방법이나 다른 패키지와 통합하는 방법의 예제가 됩니다. 가장 인기 있는 사용 사례는 문서화 목적과 라이브러리 패키지입니다.

## 패키지 컴파일하기

Move는 컴파일 언어이므로 소스 파일을 Move 바이트코드로 컴파일해야 합니다. 모듈, 멤버 및 타입에 대한 필요한 정보만 포함하고 주석과 일부 식별자(예: 상수)는 제외합니다.

이러한 기능을 시연하기 위해 _sources/hello_world.move_ 파일의 내용을 다음으로 바꿔봅시다:

```move
/// 명명된 주소 `hello_world` 아래의 모듈 `hello_world`.
/// 명명된 주소는 `Move.toml`에 설정되어 있습니다.
module hello_world::hello_world;

// 표준 라이브러리에서 `String` 타입을 임포트합니다
use std::string::String;

/// "Hello, World!"를 `String`으로 반환합니다.
public fun hello_world(): String {
    b"Hello, World!".to_string()
}
```

컴파일 중에 코드가 빌드되지만 실행되지는 않습니다. 컴파일된 패키지는 다른 모듈이나 트랜잭션에서 호출할 수 있는 함수만 포함합니다. [개념](./../concepts) 챕터에서 이러한 개념을 설명하겠습니다. 지금은 _sui move build_를 실행할 때 어떤 일이 일어나는지 봅시다.

```bash
# `hello_world` 폴더에서 실행
$ sui move build

# 또는 폴더로 이동하지 않았다면
$ sui move build --path hello_world
```

콘솔에 다음 메시지가 출력되어야 합니다.

```plaintext
UPDATING GIT DEPENDENCY https://github.com/MystenLabs/sui.git
INCLUDING DEPENDENCY Bridge
INCLUDING DEPENDENCY DeepBook
INCLUDING DEPENDENCY SuiSystem
INCLUDING DEPENDENCY Sui
INCLUDING DEPENDENCY MoveStdlib
BUILDING hello_world
```

컴파일 중에 Move 컴파일러는 자동으로 build 폴더를 생성하여 가져온 모든 의존성과 컴파일된 의존성, 그리고 현재 패키지 모듈의 바이트코드를 배치합니다.

> Git과 같은 버전 관리 시스템을 사용하는 경우 build 폴더는 무시되어야 합니다. 예를 들어, `.gitignore` 파일을 사용하고 `build`를 추가해야 합니다.

## 테스트 실행하기

테스트에 들어가기 전에 테스트를 추가해야 합니다. Move 컴파일러는 Move로 작성된 테스트를 지원하고 실행 환경을 제공합니다. 테스트는 소스 파일과 `tests/` 폴더 모두에 배치할 수 있습니다. 테스트는 `#[test]` 속성으로 표시되고 컴파일러에 의해 자동으로 검색됩니다. [테스팅](./../move-basics/testing) 섹션에서 테스트에 대해 자세히 설명합니다.

`tests/hello_world_tests.move`의 내용을 다음 내용으로 바꾸세요:

```move
#[test_only]
module hello_world::hello_world_tests;

use hello_world::hello_world;

#[test]
fun test_hello_world() {
    assert!(hello_world::hello_world() == b"Hello, World!".to_string(), 0);
}
```

여기서 우리는 `hello_world` 모듈을 임포트하고, 그것의 `hello_world` 함수를 호출하여 출력이 실제로 "Hello, World!" 문자열인지 테스트합니다. 이제 테스트가 준비되었으니 패키지를 테스트 모드에서 컴파일하고 테스트를 실행해봅시다. Move CLI는 이를 위한 `test` 명령어를 가지고 있습니다:

```bash
$ sui move test
```

출력은 다음과 유사해야 합니다:

```plaintext
INCLUDING DEPENDENCY Bridge
INCLUDING DEPENDENCY DeepBook
INCLUDING DEPENDENCY SuiSystem
INCLUDING DEPENDENCY Sui
INCLUDING DEPENDENCY MoveStdlib
BUILDING hello_world
Running Move unit tests
[ PASS    ] 0x0::hello_world_tests::test_hello_world
Test result: OK. Total tests: 1; passed: 1; failed: 0
```

패키지 폴더 밖에서 테스트를 실행하는 경우 패키지 경로를 지정할 수 있습니다:

```bash
$ sui move test --path hello_world
```

문자열을 지정하여 한 번에 하나 또는 여러 테스트를 실행할 수도 있습니다. 문자열을 포함하는 모든 테스트 이름이 실행됩니다:

```bash
$ sui move test test_hello
```
